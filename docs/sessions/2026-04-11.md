### Server-side trait detection
- New `src/aiproxy/core/traits.py` with `detect_request_traits(req_body)`
  returning `{request_has_image, request_has_file, response_is_json}`.
  Covers OpenAI / OpenRouter `image_url` / `input_image` / `file` /
  `input_file`, Anthropic `image` / `document`, the responses API `input`
  field, `response_format.{json_object,json_schema}`, and non-empty `tools`
  arrays (implies structured JSON output).
- Three new nullable `Integer` columns on the `requests` row
  (NULL = unknown / predates migration).
- `backfill_request_traits(session, rescan_all=False)` walks rows in cursor
  batches, idempotent, commits per batch.
- **Version-gated rescan:** `DETECTOR_VERSION` constant in `traits.py` is
  compared against `config["traits.detector_version"]` on every startup. On
  mismatch the full table is re-scanned and the version bumped. Bumping the
  constant is the only action required to propagate a detection-rule change
  to historical data.
- Lifespan logs `trait detector v<N>: rescanned <N>` or
  `backfilled traits for <N> requests`.

## Key decisions

1. **Trait detection happens at `forward()` time on the rewritten body**,
   not at finalize. Gives us the flags before any error path can skip them,
   and ensures what's persisted matches what the upstream actually saw after
   `Provider.rewrite_request_body`.
2. **Nullable `Integer` columns, not `Boolean`**, so we can distinguish
   "never classified" (NULL) from "classified as False" (0). The serializer
   maps both to frontend `false` via `bool(row.x)`.
3. **`DETECTOR_VERSION` + config table** instead of a migration script for
   detector-rule changes. One constant bump → automatic full rescan on next
   boot, with a single log line to confirm. Scales to unlimited future rule
   additions with zero plumbing per change.
4. **Dropped retroactive backfill as a CLI script**. Originally considered,
   but startup-driven is simpler UX and doesn't require the user to remember
   to run something after pulling.
5. **SVG `mask-image` for trait icons** instead of inline `<svg>` elements
   per row. Zero DOM bloat per row, CSS controls color via
   `background-color`, and the shared `.trait-icon` class centralizes sizing.
6. **All 4 trait icons normalized to the same 12×12 viewBox with content in
   `y ∈ [1.5, 10.5]`**. This was the fix for the visual height mismatch
   the user flagged — earlier versions had heights ranging from 8 to 10
   units, making the row look ragged.
7. **Infinite scroll instead of true virtualization**. Variable row heights
   (label pills) make real virtual lists painful in vanilla JS for marginal
   benefit at the dataset sizes we actually hit. Documented the trade-off
   in CLAUDE.md.
8. **Explicitly did NOT touch the login page**. Scope discipline — theme
   system is a dashboard feature for this PR.

## Churn episodes — lessons

### Episode 1: The "empty lines in tool_use preview" fix (pre-compact)
- **What happened:** Two rounds. First fix targeted CSS padding / line-height
  on the assumption it was spacing. User came back with a second screenshot
  circling *literal* blank lines between `<div>` children.
- **Root cause:** `.preview-content { white-space: pre-wrap }` made the
  newlines + indent in the JS template literal (`\n    ` between
  `</div>` and `<div>`) render as visible whitespace. The fix was moving
  `pre-wrap` off `.preview-content` onto `.preview-part.text` /
  `.preview-part.thinking` so only text-bearing parts preserved their own
  whitespace.
- **Lesson:** When a UI "spacing" complaint shows a screenshot with
  circled regions, read the *exact* pixels being circled before hypothesizing.
  CSS `white-space` interacts with template-literal formatting in ways you'll
  never spot just by looking at the rendered markup.

### Episode 2: Stream-mark glyph alignment
- **What happened:** Picked U+223F `∿` for the streaming marker. User
  screenshot showed it sitting visibly lower than the status dot and req id
  text, even with flex `align-items: center` on the parent.
- **Root cause:** Math-operator codepoints have baselines below the x-height
  of surrounding text. No amount of flex alignment fixes this — the glyph's
  metrics are wrong for inline use.
- **Fix:** Replaced with an SVG waveform via `mask-image`, which sits on the
  pixel grid and respects `vertical-align: middle`. Then replaced *that*
  with the signal-bars icon on user feedback.
- **Lesson:** Don't use unicode glyphs as inline icons for visual alignment.
  SVG mask-image is the same amount of CSS, renders pixel-perfect, and lets
  you control color via `background-color`.

### Episode 3: Trait icon visual heights unequal
- **What happened:** Shipped 4 icons with inconsistent visual heights
  (stream=9, image=8, file=10, json=9 units in a 12-unit viewBox).
- **Root cause:** Each icon was drawn independently with no shared template.
  viewBox was the same 12×12 but content y-ranges drifted.
- **Fix:** Normalized all 4 to `y ∈ [1.5, 10.5]` (9 units centered), and
  stroke-width to 1.2. Re-wrote each SVG path against the shared grid.
- **Lesson:** For multi-icon systems, define the "inner box" constraint
  *before* drawing any path. Same viewBox ≠ same visual size.

### Episode 4: The "can't see JSON icon for tool_use requests"
- **What happened:** Initial detector checked `response_format.type` only.
  User's Anthropic workbench requests all used `tools` / tool_use for
  structured output, so none got the JSON icon even though semantically
  they should have.
- **Root cause:** Anthropic has no `response_format` — tool_use *is* their
  structured output mechanism. Detector scope was wrong.
- **Fix:** Expanded `detect_request_traits` to also check `isinstance(tools,
  list) and tools`. Bumped `DETECTOR_VERSION = 2` so historical rows get
  re-scanned on next boot. Added unit + integration tests for tools path.
- **Lesson:** When classifying "structured output requests", don't assume
  `response_format` is universal. Tool/function calling is the more common
  structured-output mechanism in practice. Design detectors to be
  version-gated from day 1 so rule expansion doesn't require manual
  retroactive data cleanup.

### Episode 5: Live timer that never appeared to tick
- **What happened:** User asked for live-ticking time-ago on list rows.
  Implemented via `setInterval` + `data-ts` attribute + selective textContent
  update. User said "no change visible". Extended the per-second format to
  a 10-minute range. Still "no change". User said "delete it".
- **Root cause:** Never confirmed. Possibilities: (1) user was watching rows
  all >10min old and didn't notice ticks at the minute/hour level;
  (2) browser tab inactive throttled the interval; (3) actual bug in the
  tick logic that we never diagnosed because the user called it off.
- **Lesson:** When a user reports "feature doesn't work" and asks for
  removal after one round of fixes, don't dig for a root cause — just
  revert cleanly. But leave a note (here) so the next attempt can start
  with "what actually went wrong last time".
