## Completed work

### Dockerization

- **Multi-stage Dockerfile** (builder uses `uv sync --frozen --no-dev`,
  runtime is `python:3.12-slim` as non-root `app` user with healthcheck).
- **`compose.local.yml`** binds `127.0.0.1:8000`, bind-mounts `./data/` for
  SQLite, no TLS.
- **`compose.server.yml`** joins the external `caddy-net`, publishes no host
  ports, uses a named volume `aiproxy-data` for SQLite, sets
  `SECURE_COOKIES=true`, caps the JSON-file log driver at 100MB rolling.
- **`uvicorn --workers 1 --proxy-headers --forwarded-allow-ips "*"`** —
  single-process because SQLite file locking + in-memory StreamBus both
  assume one process; `--proxy-headers` so the cookie's `Secure` flag and
  client IP come through Caddy correctly.

### SQLite hardening

- `db/engine.py` now attaches a per-connect listener that sets
  `journal_mode=WAL`, `synchronous=NORMAL`, `busy_timeout=10000`,
  `foreign_keys=ON` on every new sqlite connection. Guarded on
  `engine.dialect.name == "sqlite"` so the listener doesn't attach to
  non-sqlite dialects — and it's a no-op on in-memory DBs (they always use
  memory journal), which keeps the 234 tests passing unchanged.

### Prometheus instrumentation

- **`src/aiproxy/metrics.py`** — seven metric families (5 counters, 2
  histograms, 1 gauge), plus a `record_completion()` helper that keeps the
  label normalization in one place.
- **Cardinality bounds**: `model` label accepted (OpenRouter could push it
  to ~100s but that's fine for a self-hosted proxy); `X-AIProxy-Labels` is
  explicitly NOT exposed as a Prometheus label since it's user-driven and
  unbounded. Per-`api_key_id` breakdown lives in the SQLite-backed requests
  list only.
- **Hooks**: `PassthroughEngine._finalize` for the terminal state (done /
  error / canceled); the two pre-stream failure paths (ConnectError /
  ReadTimeout) also emit with `error_class` set. `RequestRegistry`
  start/finish inc/dec `aiproxy_active_requests`.
- `/metrics` route served unauthenticated in Prometheus text format.

### Grafana — file-based provisioning

- `deploy/grafana/provisioning/datasources/prometheus.yaml` declares the
  Prometheus datasource with `uid: prometheus` + a `deleteDatasources`
  block to clear any UI-created row that would otherwise block the UID
  swap.
- `deploy/grafana/provisioning/dashboards/provider.yaml` with

## Churn episodes

### 1. UV_HTTP_TIMEOUT — wasted one build iteration

**What happened:** First `docker compose build` on the server failed
because `websockets==16.0` and `certifi==2026.2.25` downloads timed out at
uv's default 30s. The symptom looked like "slow network", but when I
tested from a fresh `python:3.12-slim` container on the same host, PyPI
returned in 282ms. So it wasn't network — it was a transient PyPI
edge-node hiccup that the default 30s couldn't absorb.

**Fix:** `ENV UV_HTTP_TIMEOUT=300` in the Dockerfile (one-line commit
`ce373a3`). The user correctly pushed back ("这是新加坡服务器, 不用该
遇到这些问题") which forced me to actually diagnose rather than paper
over it — the 300s timeout is the right fix regardless, but the
investigation confirmed there's no systemic network problem to worry
about.

**Lesson:** When a build fails with "timeout", always verify whether the
network is actually slow before cargo-culting a timeout bump. The bump is
still good defense-in-depth, but the investigation is what tells you
whether there's a deeper problem.

### 2. Grafana datasource UID — two rounds of fixes

**What happened:** Shipped `prometheus.yaml` without an explicit `uid:`
field, so Grafana auto-generated `PBFA97CFB590B2093`. The dashboard JSON
references `uid: prometheus`, so every panel would have rendered "data
source not found". Added `uid: prometheus` to the YAML (`614fe33`) —
Grafana then crashed on restart with `"Datasource provisioning error:
data source not found"` because a UI-created Prometheus row already
occupied the name. Had to add `deleteDatasources` block to wipe the
pre-existing row before insertion (`3d70f9a`).

**Lesson:** When provisioning a datasource under a pinned UID for the
first time against an already-initialized Grafana, always pair
`datasources:` with `deleteDatasources:` by name. UI-created rows don't
go away just because a config file shows up.

### 3. Simplifier catching 4 factual errors in operations.md

**What happened:** Wrote the operations doc working from memory of what
the code does. The simplifier agent caught:
- `sqlite3` CLI isn't in `python:3.12-slim` — backup recipe would've failed
- claimed client API keys are not stored plaintext — they are
- claimed Prometheus metrics slice by `api_key_id` — they don't (cardinality)
- Caddy `handle_path` vs `handle` confusion for the `/metrics` 404 recipe

**Lesson:** Even "obvious" documentation benefits from an independent pass.
Three of the four errors were confident-sounding claims I'd have been
embarrassed to leave in a runbook. The simplifier step of `/ship` pulled
its weight on this one.

## Decisions worth flagging

### Plan B over A/C for server migration

Three options considered:
- **A:** Deploy as a new stack alongside old, different subdomain
- **B:** Stop old stack entirely, reuse `ai.sat1600ap5.online`
- **C:** Keep old running, rewire DNS

Chose B because the old `openai-proxy` stack was unused. Benefits:
existing Caddy site `002-bollegecoard.caddy` (`ai.sat1600ap5.online →
aiproxy:8000`) worked zero-config as long as we kept the service name
`aiproxy`. Trade-off: if the new deploy had failed, there's no fallback
to the old. We judged this safe because the old stack had been unused
for weeks.

### Single dashboard, 11 panels, not three dashboards

Resisted the urge to split into Overview / Latency / Cost. One
single-page dashboard fits the "quick deploy, enough visibility" frame
and avoids the context-switch penalty between three pages. Easy to split
later if panels get crowded.

### Provisioning files in ai-proxy repo (not logs-stack)

The Grafana provisioning YAML + JSON live under `deploy/grafana/` in
this repo, not in the logs-stack compose dir. Reasoning: observability
config is tightly coupled to the app's metric names. When you rename
`aiproxy_requests_total` you want the dashboard change to land in the
same PR. Logs-stack gets only a one-line bind-mount reference.

### Skipping the heavyweight brainstorm

User invoked `/superpowers:brainstorming` then pulled back — "别搞那些
复杂得了, 追求快速部署". Compressed the design into a single paragraph
("7 metrics, 1 dashboard, 11 panels, provisioning-based, drop Tier 2+3"),
got approval, executed. This was the right call for a single-person
self-hosted deployment.
